# 附錄G - 編碼代理(Coding Agents)

## 氛圍編碼(Vibe Coding)：起步之道

「氛圍編碼」(vibe coding) 已成為推動快速創新和創意探索的強大技巧。這種做法利用大型語言模型(LLMs) 生成初稿、勾勒複雜邏輯或建立快速原型，大幅降低起步時的阻力。它能有效解決「空白頁」問題，讓開發者迅速由模糊概念過渡到具體、可執行的程式碼。當探索陌生的應用程式介面(APIs) 或測試新穎的架構模式(architectural patterns) 時，氛圍編碼尤其奏效，因為它免除了立即追求完美實作的需要。生成的程式碼往往成為創意的催化劑，為開發者提供批判、重構和擴展的基礎。其主要優勢在於加速軟件生命週期(software lifecycle) 初期的探索與構思階段。然而，儘管氛圍編碼擅長腦力激盪，要打造穩健、可擴展並易於維護的軟件，仍需採用更有結構的方式，從純生成過渡到與專門編碼代理(coding agents) 的協作夥伴關係。

## 代理(Agents) 作為團隊成員

雖然最初的浪潮集中在原始程式碼生成——非常適合構思的「氛圍程式碼」——業界現正轉向適用於生產工作的更整合、更強大的典範。最有效率的開發團隊並非只是把任務外判給代理，而是用一套精密的編碼代理來強化自己。這些代理如同不知疲倦的專才團隊成員，放大人類創意，並大幅提升團隊的可擴展性與速度。

這場演進亦反映在業界領袖的言論中。2025年初，Alphabet 行政總裁 Sundar Pichai 表示，在 Google，**"***over 30% of new code is now assisted or generated by our Gemini models, fundamentally changing our development velocity."** Microsoft 亦提出相同論點。這股產業潮流顯示真正的前沿不在於取代開發者，而是賦能開發者。目標是一種增強式關係：人類負責掌舵架構願景與創意解難，代理則處理專門且可擴展的任務，例如測試、文件撰寫和審查。

本章提出一套組織人類—代理團隊的框架，核心理念是人類開發者擔任創意主導與架構師，而人工智能代理則作為戰力倍增器(force multipliers)。此框架建立在三項基本原則之上：

1. **人類主導的協調(Human-Led Orchestration)：** 開發者是團隊領袖兼項目架構師。他們必須全程參與，統籌工作流程、訂立高層目標並作出最終決策。代理固然強大，但只是支援性的合作者。開發者決定啟用哪個代理、提供所需背景，並最重要地，對任何代理生成的輸出行使最終判斷，確保成果符合項目的質量標準與長遠願景。
2. **上下文至上(The Primacy of Context)：** 代理的表現全賴背景資訊的質與量。再強大的大型語言模型若缺乏完善上下文也無用。因此，框架強調由人類主導的細緻上下文整理。避免使用自動化、黑箱式的上下文擷取。開發者需為其代理團隊成員準備完備的「簡報」，包括：
   * **完整的程式碼庫(The Complete Codebase)：** 提供所有相關源碼，讓代理理解現有模式與邏輯。
   * **外部知識(External Knowledge)：** 提供特定文件、應用程式介面定義或設計文件。
   * **人類簡報(The Human Brief)：** 清楚表述目標、需求、拉取請求(pull request) 描述及風格指南。
3. **直接模型存取(Direct Model Access)：** 要達致最尖端的成果，代理必須能直接存取前沿模型(frontier models)（例如 Gemini 2.5 PRO、Claude Opus 4、OpenAI、DeepSeek 等）。使用較弱的模型或透過截斷上下文的中介平台，只會削弱表現。此框架旨在建立人類主導與底層模型原生能力之間最純粹的對話，確保每位代理都能發揮最大潛能。

此框架將團隊結構化為一組專業代理，各司開發生命週期(development lifecycle) 的核心職能。人類開發者扮演中央協調者，分派任務並整合成果。

## 核心組成(Core Components)

為有效利用前沿大型語言模型，此框架為專業代理團隊分配明確的開發角色。這些代理並非獨立應用程式，而是透過精心設計的角色導向提示(prompts) 與上下文，在大型語言模型內召喚的概念化人格。此方法可確保模型的龐大能力能精準聚焦於手頭任務，從撰寫初稿到進行細膩、嚴謹的審查。

**總指揮(Orchestrator)：人類開發者(Human Developer)：** 在這個協作框架中，人類開發者擔任總指揮，是整個代理團隊的核心智慧與最終權威。

* **角色(Role)：** 團隊領袖、架構師與最終決策者。總指揮負責界定任務、準備上下文並核實代理完成的所有工作。
  * **介面(Interface)：** 開發者自用的終端機(terminal)、編輯器(editor) 以及所選代理原生的網頁介面(web UI)。

**上下文預備區(Context Staging Area)：** 作為任何成功代理互動的基礎，上下文預備區是人類開發者細緻準備完整且符合任務需求簡報的地方。

* **角色(Role)：** 為每項任務提供專屬工作區，確保代理收到完整且準確的簡報。
  * **實作方式(Implementation)：** 使用一個臨時目錄(temporary directory)（如 task-context/），內含記錄目標的Markdown 文件、程式碼檔案與相關文件。

**專業代理(The Specialist Agents)：** 透過針對性的提示，我們可建立一支專業代理團隊，每位代理都針對特定開發任務量身訂造。

* **腳手架代理(Scaffolder Agent)：實作者(The Implementer)**
  * **目的(Purpose)：** 根據詳細規格撰寫新程式碼、實作功能或建立樣板。
    * **調用提示(Invocation Prompt)：** 「你是一位資深軟件工程師(senior software engineer)。請根據 01_BRIEF.md 的需求以及 02_CODE/ 中現有的模式來實作該功能……（"You are a senior software engineer. Based on the requirements in 01_BRIEF.md and the existing patterns in 02_CODE/, implement the feature..."）」
  * **測試工程師代理(Test Engineer Agent)：品質守護者(The Quality Guard)**
    * **目的(Purpose)：** 為新的或現有的程式碼撰寫全面的單元測試(unit tests)、整合測試(integration tests) 與端到端測試(end-to-end tests)。
    * **調用提示(Invocation Prompt)：** 「你是一位品質保證工程師(quality assurance engineer)。請針對 02_CODE/ 中的程式碼，使用［測試框架，例如 pytest］撰寫完整的單元測試套件，涵蓋所有邊界情況，並遵循項目的測試理念。（"You are a quality assurance engineer. For the code provided in 02_CODE/, write a full suite of unit tests using [Testing Framework, e.g., pytest]. Cover all edge cases and adhere to the project's testing philosophy."）」
  * **文件撰寫代理(Documenter Agent)：書記(The Scribe)**
    * **目的(Purpose)：** 為函式(functions)、類別(classes)、應用程式介面(APIs) 或整個程式碼庫生成清晰簡潔的文件(documentation)。
    * **調用提示(Invocation Prompt)：** 「你是一位技術寫手(technical writer)。請為所提供程式碼中定義的應用程式介面生成Markdown 文件，包含請求/回應範例，並解釋每個參數。（"You are a technical writer. Generate markdown documentation for the API endpoints defined in the provided code. Include request/response examples and explain each parameter."）」
  * **優化代理(Optimizer Agent)：重構夥伴(The Refactoring Partner)**
    * **目的(Purpose)：** 提出效能優化(performance optimizations) 與程式碼重構(code refactoring)，以提升可讀性、可維護性與效率。
    * **調用提示(Invocation Prompt)：** 「請分析所提供的程式碼，尋找效能瓶頸(performance bottlenecks) 或可為清晰度而重構之處。提出具體變更，並解釋為何能帶來改善。（"Analyze the provided code for performance bottlenecks or areas that could be refactored for clarity. Propose specific changes with explanations for why they are an improvement."）」
  * **流程代理(Process Agent)：程式監督者(The Code Supervisor)**
    * **批判(Critique)：** 該代理會先進行初步審查，識別潛在錯誤、風格違規和邏輯缺陷，如同靜態分析工具(static analysis tool)。
    * **反思(Reflection)：** 然後分析自身批判，綜合發現、優先處理最關鍵問題，並剔除瑣碎或低影響建議，向人類開發者提供高層次且可行的摘要。
    * **調用提示(Invocation Prompt)：** 「你是一位首席工程師(principal engineer)，正在進行程式碼審查(code review)。首先，請對變更作出詳細批判。其次，請反思你的批判，提供一份精簡且排序清晰的重點回饋摘要。（"You are a principal engineer conducting a code review. First, perform a detailed critique of the changes. Second, reflect on your critique to provide a concise, prioritized summary of the most important feedback."）」

最終，這種由人類領導的模式為開發者的戰略指導與代理的戰術執行帶來強大協同效應。結果，開發者可以超越日常瑣事，將專長聚焦於帶來最大價值的創意與架構挑戰。

## 實務落地(Practical Implementation)

### 設定清單(Setup Checklist)

要有效落實人類—代理團隊框架，建議採取以下設定，兼顧掌控力與效率提升。

1. **取得前沿模型存取權(Provision Access to Frontier Models)：** 為至少兩款頂尖大型語言模型取得安全應用程式介面金鑰(API keys)，例如 Gemini 2.5 Pro 與 Claude 4 Opus。這種多供應商策略可進行比較分析，亦能避開單一平台的限制或停機。這些憑證必須像其他生產機密一樣妥善管理。
2. **實作本地上下文協調器(Implement a Local Context Orchestrator)：** 與其使用臨時腳本，不如使用輕量指令列工具(CLI tool) 或本地代理執行器(agent runner) 來管理上下文。這些工具應允許你在項目根目錄(root) 定義簡單的設定檔（例如 context.toml），指定要整合到大型語言模型提示中的檔案、目錄甚至網址(URLs)。如此可確保你完全且透明地掌控每次請求中模型所見的內容。
3. **建立版本控制的提示庫(Establish a Version-Controlled Prompt Library)：** 在項目的Git 儲存庫(Git repository) 中建立專用的 /prompts 目錄。將各專業代理的調用提示（例如 reviewer.md、documenter.md、tester.md）以Markdown 檔案形式存放其中。將提示視為程式碼，可讓整個團隊共同協作、精煉並為給予人工智能代理的指示進行版本控制。
4. **將代理工作流程整合至Git 掛鉤(Integrate Agent Workflows with Git Hooks)：** 透過本地Git 掛鉤(git hooks) 自動化你的審查節奏。例如，可設定預先提交(pre-commit) 掛鉤，在提交前自動觸發審查代理(Reviewer Agent)，於終端機直接呈現其批判與反思摘要，讓你在完成提交前即獲得即時回饋，把品質保證步驟嵌入開發流程。

### 帶領增強團隊的原則(Principles for Leading the Augmented Team)

成功帶領此框架，需要從單兵作戰者，進化為人類—人工智能團隊的領袖，遵循以下原則：

* **維持架構主導(Maintain Architectural Ownership)：** 你的角色是設定策略方向與掌握高層架構。你界定「做什麼」與「為什麼」，再以代理團隊加速「如何去做」。你是設計的最終裁判(arbiter)，確保每個組件都符合項目的長遠願景與質量標準。
* **精通簡報藝術(Master the Art of the Brief)：** 代理輸出的質素直接反映輸入的質素。為每項任務提供清晰、毫不含糊且全面的上下文，掌握簡報的藝術。把你的提示視為為新加入且能力出眾的團隊成員準備的完整簡報套件，而非一句命令。
* **成為終極品質關卡(Act as the Ultimate Quality Gate)：** 代理輸出永遠只是提案，絕非命令。視審查代理(Reviewer Agent) 的回饋為強大訊號，但你才是最終品質關卡。運用你的領域專業與項目知識去驗證、質疑並核准所有變更，守護程式碼庫的完整性。
* **進行迭代式對話(Engage in Iterative Dialogue)：** 最佳成果源於對話而非獨白。若代理的初步輸出未如理想，不要全盤否定——請加以調整。提供修正回饋、補充上下文，再促成另一輪嘗試。這種迭代式對話至關重要，尤其是與審查代理互動時；其「反思」輸出旨在引發協作討論，而非僅僅提交最終報告。

## 結語(Conclusion)

未來的程式開發已然來臨，而且是增強式的。孤身開發者的年代已經過去，如今的典範是開發者帶領專門的人工智能代理團隊。這個模式並未削弱人類角色，反而透過自動化日常工作、擴大個人影響力以及達致前所未有的開發速度來提升人類價值。

透過把戰術執行交給代理，開發者得以把心力專注於真正重要之處：策略創新、韌性架構設計以及為打造深受用戶喜愛產品所需的創意解難。人機關係的本質已被重新定義，不再是人類對抗機器，而是人類創意與人工智能結合成一支無縫整合的團隊。
